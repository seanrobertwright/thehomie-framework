# R1 Adversarial Review — CLUTCH v3 Pre-Build Gate

> Template — orchestrator substitutes `{PRD_PATH}`, `{PRP_PATH}`, `{PHASE_N}`, `{PROJECT_PATH}`, `{PRD_NAME}` before piping to `codex exec`.

You are the Round 1 adversarial reviewer for **Phase {PHASE_N}** of **{PRD_NAME}**. The PRP was generated by a fresh research agent in CLUTCH v3 Step 1 with full codebase analysis context. Your job is to find class-of-bugs in the design BEFORE implementation begins. Be sharp. Push back. Be specific about file paths, line numbers, and failure modes.

This is the standard adversarial-review shape proven on PRP-7 (Apr 2026): identify blockers (must-fix), majors (should-fix), and minors (nice-to-have). Your output becomes input to a revision pass, then R2 verifies the revision was real.

## Read these first (in order)

1. `{PRD_PATH}` — the parent PRD; understand the strategic context and where this phase fits in the larger architecture.
2. `{PRP_PATH}` — the just-generated PRP for Phase {PHASE_N}; this is what you are reviewing.
3. `{PROJECT_PATH}/PRPs/planning/{PRD_NAME}-phase-{PHASE_N}-analysis.md` — the codebase analysis the research agent produced. Verify the PRP claims are consistent with what the analysis found.
4. `{PROJECT_PATH}/vault/memory/MEMORY.md` (Reference → Global Rules section) — the three anti-pattern rules that apply to all framework code.
5. Any project-specific governance: `{PROJECT_PATH}/AGENTS.md`, `{PROJECT_PATH}/CLAUDE.md` — repo-level conventions and constraints.

## Three Mandatory Anti-Pattern Rules

Verify the PRP does NOT introduce violations of these:

1. **Rule 1 — No tunable config in default args.** Use `None` sentinel, resolve in body. Particular concern: helpers shared across entry points must not freeze config values at import time.
2. **Rule 2 — Meta is derived state, not source of truth.** Anywhere a guard gates a destructive operation, the guard MUST inspect physical backend state, not a config/meta value.
3. **Rule 3 — Optional-provider calls via flag-checked helper.** Module-attribute lookup (`from runtime import langfuse_setup; langfuse_setup.is_langfuse_enabled()`) so monkeypatch propagates.

## Repo-faithfulness audit

Per the codebase, verify that EVERY claim in the PRP about existing files, line numbers, function signatures, or behavior is grounded in actual code. Open the files cited and confirm. Fabricated repo claims are blockers.

## Inventory completeness

If the PRP claims to enumerate something (entry points to modify, hardcoded paths to refactor, services to consolidate, schemas to migrate), verify completeness by independently scanning the repo. List anything the PRP missed as a blocker.

## Boot-time / import-order concerns

If the PRP changes any module that imports `config`, `runtime`, or other framework foundations: walk through the import order and verify caching constants resolve correctly. Cite file:line for the imports you traced.

## Anti-pattern coverage of the PRP's test plan

Check that the PRP's tests actually exercise the target behavior — not path math, not file-presence checks, not commands that succeed for any non-empty string. Tests that "pass without exercising the target" are gas-station tests; flag them as majors.

## Adversarial checklist coverage

If the PRP lists adversarial concerns, verify each item:
- Is realistic (covers a class-of-bug, not a one-off)
- Has a corresponding test or grep
- Is not duplicated or paraphrased from another item
