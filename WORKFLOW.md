### Phase 2 Key Architectural Decisions

1. **`LifecycleError(RuntimeError)`** — operator-facing exception subclass. Subclassing `RuntimeError` keeps existing `except RuntimeError:` catches matching, but the framework convention is: operator-visible failures use `LifecycleError`; bare `RuntimeError` indicates a bug and should propagate. Click handler narrows catch to `(LifecycleError, ValueError, FileExistsError, FileNotFoundError)`. R3 NNM3.
2. **`_try_acquire_state_lock` mirrors `shared.file_lock` byte-0 semantics** — opens with `"r+"` (no truncate, FP=0); on Windows `msvcrt.locking(fd, LK_NBLCK, 1)` locks byte 0; on POSIX `fcntl.flock(fd, LOCK_EX | LOCK_NB)` whole-file. Pass-2 had this wrong with `"a"` mode (FP=EOF) producing false-free readings on Windows. R3 NNB1.
3. **Subprocess holder `sys.path` insertion** — `tests/_holders/hold_state_lock.py` does `sys.path.insert(0, str(Path(__file__).resolve().parents[2]))` BEFORE `import shared` because the child Python process does not execute pytest's `conftest.py`. R3 NNB2. Plus a separate `hold_byte0_winlock.py` helper for the byte-0 trap regression (uses `msvcrt` directly without `import shared`, so it doesn't truncate the lock file). R4 NNNM1.
4. **Containment + symlink check FIRST** — `delete_profile()` resolves `_profile_root(name)`, validates containment via `is_relative_to(_profiles_root().resolve())`, and checks `not is_symlink()` BEFORE entering `_acquire_delete_lock` context. R2 NM4.
5. **Wrapper template path quoting/escaping** — `shlex.quote` for POSIX shell; `_cmd_escape` returns unquoted value (cmd template uses `set "HOMIE_HOME=<escaped>"` — quote wraps whole assignment); `_ps_single_quote` for PowerShell; `plistlib.dumps()` for launchd (NOT hand-built XML); systemd quotes BOTH `Environment=` AND `WorkingDirectory=`. R3 NNM2.
6. **`_acquire_delete_lock` is sentinel-file mutex, not advisory OS lock** — uses `open(path, 'x')` (O_EXCL); the FILE'S EXISTENCE is the guard, fd closed immediately. Distinguished from `_try_acquire_state_lock` which IS advisory-OS-lock-based. R4 minor.
7. **State-lock probe runs FIRST in `quiesce_profile`** (post-build F1) — if any process holds a state lock, raise `LifecycleError` BEFORE bot kill or unit uninstall. Prior order (lock check after side effects) could leave a half-quiesced profile when the bot was SIGTERM'd and launchd/systemd disabled, then deletion refused. State-machine table in PRP §1955-2007 is now canonical.
8. **`export_profile("default")` stages from `get_default_paths()` keys** (post-build F2) — does NOT recursively `copytree()` the install repo. `_stage_default_export_tree()` maps SAFE keys (`memory`, `state`, `logs`, `run`, `archon`, `home`, `cron`, `sessions`, `skills`) into a profile-shaped staging tree; `env_file`, `credentials`, `workspace`, `data`, `integrations`, `.git` are explicitly excluded. Post-stage `_assert_no_secrets_in_staged_tree()` walks and raises if any secret-shaped path survived (defense in depth + fail closed).
9. **Secret stripping fails CLOSED** (post-build F3) — `_copytree_with_strip` raises if `.env` rewrite fails when `carry_secrets=False`. Export uses pre-stage ignore filter + post-stage scan instead of best-effort post-unlinks. On Windows readonly attributes/ACLs that previously left secrets in the archive now raise instead.
10. **Public clone CLI routes through `create_profile`** (post-build F4) — `profile clone` and `profile clone-all` Click handlers call `create_profile(dst, clone=True/clone_all=True, clone_from=src, ...)` rather than `clone_profile()` directly. This means clone destinations get the full `_REQUIRED_PROFILE_DIRS` / `_REQUIRED_MEMORY_DIRS` / `_REQUIRED_IDENTITY_FILES` backfill AND a wrapper alias. Library `clone_profile()` helper still exists with bare-helper semantics; zero production callers per `rg`; not exported in `personas.__all__`.
11. **`source_memory_dir` adapter for `clone_from="default"`** (post-build F5 + NF1) — when source is `"default"`, `create_profile` resolves source memory from `get_default_paths()["memory"]` (the actual `<install>/vault/memory/` location) rather than expecting `<source_root>/memory/`. ### Phase 4 Key Architectural Decisions

1. **Shared `_assert_source_column_shape` helper (R3 NNB1)** — module-level helper called by BOTH SQLite and Postgres migration paths so the two backends cannot drift on shape requirements (TEXT/text type, NOT NULL, default contains "interactive", zero NULL/empty rows). Forces post-check to ALWAYS run (whether ALTER executed or skipped) — closes the R3 "presence-only skip" failure mode.
2. **`_run_source_migration(_alter_executor=)` test seam (R3 NNM2)** — SQLite migration factored so the ALTER can be replaced by an injected callable in tests. Avoids monkeypatching `sqlite3.Connection.execute` (immutable in Python 3.14). Production path uses `cursor.execute` default; tests pass a callable that simulates concurrent-ALTER duplicate-column races.
3. **Broadened race classifier (post-build F2)** — `_is_duplicate_source_column_error()` accepts both `"duplicate column"` AND `"already exists"` + `"source"`/`"column"` patterns case-insensitively. Hardens against SQLite version drift in error message wording. Other `OperationalError`s (locked DB, syntax, missing table) still propagate.
4. **Set-once invariant on `Session.source`** — `create()` writes `normalize_source(session.source)`; UPDATE branches in engine/router/core_handlers explicitly do NOT touch source. Resume inherits the existing row's source. Verified across 5 writer paths via tests + `existing.source = ...` grep returning 0 matches.
5. **PRD §7.10 wins over §9.3 (R1 M6)** — `SOURCE_HIDDEN_BY_DEFAULT = ("tool", "hook")` with `cron` VISIBLE by default. Documented as canonical decision; widens Hermes' `("tool",)` to include `hook` since The Homie has 5 first-class adapters and hook-driven sessions could outnumber interactive 5:1.
6. **DB-column over env-var (Hermes deviation)** — Hermes uses `HERMES_SESSION_SOURCE` env at session-create time; The Homie stores in `chat_sessions.source` column. Trade-off accepted: column requires migration (idempotent + physical-state-checked), but `session list` filtering by env var would require scanning every running process — not viable. Documented as deliberate deviation, not Hermes-faithful.
7. **`"unknown"` over `"default"` for resolver fallback (R3 NNM1)** — `_resolve_active_profile_name()` returns `"unknown"` on resolver Exception, NOT `"default"`. Misattributing a profile-resolution failure to the actual default profile would silently misfile failures downstream. Module-attribute lookup of `personas.get_active_profile_name` so test monkeypatches propagate (Rule 3).
8. **CLI exception envelope preserves `--source` (post-build F1)** — `build_quiet_error_envelope(exc, *, source="interactive")` accepts the parsed Click flag value. PRD §7.11 requires JSON `source` field to echo `--source` or `interactive`; iteration-1 fix routed `cli.py:199-202` to pass `source=source` so a Paperclip-style `thehomie chat --source tool -q ... -Q` that fails during engine setup is correctly classified as `tool`, not silently relabeled `interactive`.

### ARCHON_HOME Architecture (the right design — discovered AFTER R3)

**Primary sources**: archon.diy/reference/configuration + Cole Medin's Archon README at github.com/coleam00/archon (also referenced in `vault/memory/reference_archon_repo.md`)

Archon honors `ARCHON_HOME` env var (default `~/.archon`). Workflow discovery is **3-layer with override**:
1. Bundled defaults (17 workflows shipped with the binary)
2. Home-scoped: `$ARCHON_HOME/workflows/` (overrides bundled)
3. Repo-scoped: `<cwd>/.archon/workflows/` (overrides both)

The runner pattern that ACTUALLY works (cwd is the thehomie checkout — a git repo, satisfying Archon's git-repo requirement):

```bash
# thehomie -p sales archon run profile-isolation-smoke
ARCHON_HOME=$HOMIE_HOME/.archon \
  archon workflow run profile-isolation-smoke \
  --cwd $ARCHON_SOURCE_REPO \
  --no-worktree   # for the smoke; real workflows can use Archon's worktree management
```

What this gets you per-profile (automatically, by ARCHON_HOME redirection):
- Workflows: `$HOMIE_HOME/.archon/workflows/` (overrides bundled; profile-customizable)
- DB: `$HOMIE_HOME/.archon/archon.db` (Archon's SQLite, distinct per profile)
- Worktrees: `$HOMIE_HOME/.archon/worktrees/` (Archon-managed; NO manual `git worktree add` in YAML needed)
- Workspaces: `$HOMIE_HOME/.archon/workspaces/`
- Logs: `$HOMIE_HOME/.archon/logs/`
- State: `$HOMIE_HOME/.archon/state/`

Per-profile isolation is **automatic**. The PRD §16 row 6 acceptance test asserts:
- `$HOMIE_HOME_sales/.archon/archon.db` exists AND `$HOMIE_HOME_engineering/.archon/archon.db` exists AND they're distinct files
- `$HOMIE_HOME_sales/.archon/workspaces/...` populated, distinct from engineering's
- Smoke YAML's bash node writes a marker file with `${HOMIE_NAME}` to `$HOMIE_HOME/.archon/artifacts/profile-marker.txt` — assert distinct content per profile

Smoke YAML simplification: no need for manual `git worktree add` machinery (Archon does it). Just bash nodes that write profile-marker files using `$HOMIE_HOME` + `$HOMIE_NAME` env. ~15 LOC instead of the current ~60 LOC.
